# DEVELOPMENT_RULES.md

このファイルは、コード変更・削除・共通化・デプロイの際に毎回従うべき作業ルールをまとめたものである。

CLAUDE.mdのルールに次いで優先して参照すること。

---

## 制定の経緯

2026-07-04、生年月日入力UIの横断レビューでguardian.php/zense.phpの
`.date-row select.form-input{flex:1;background-color:#1e1738;-webkit-appearance:none;appearance:none}`
から「比率を壊していた`flex:1`」だけを消すつもりが、同じルールに同居していた
`background-color`と`appearance:none`まで巻き添えで削除してしまった。

その結果、本番デプロイ後にguardian/zenseの生年月日プルダウンが白背景・矢印重複になる事故が発生した。

このルールは、この種の「巻き添え削除」を二度と起こさないための義務事項である。

---

## 役割：レビュー担当（実装者ではない）

デフォルトの役割は「実装者」ではなく「コードレビュー担当」である。

ユーザーが明確に**「実装してください」**と指示するまで、コードの変更・削除・追加は一切行わない。
「修正しました」「反映しました」「同期してください」「実装しました」という表現は、
明示の実装指示が出るまで使わない。

実装速度よりレビュー品質を優先する。1回で終わらせようとしない。レビューだけで複数回
やり取りしてよい。変更したくなったら、まず「変更しなくてもよい理由」を検討する。

## 作業フロー（STEP1〜6、必須の順序）

### STEP1：変更対象のコード全体を確認する
対象の関数・CSS・コンポーネント全体、同じセレクタ・同じ関数・同じコンポーネントを
利用している箇所、関連する共通コンポーネントまで確認する。部分だけ見て判断しない。

### STEP2：影響範囲を洗い出す
最低限：見た目／JavaScript／PHP／レスポンシブ／hover／focus／ブラウザ差異／
他ページ／共通コンポーネント。

### STEP3：変更内容を整理する
【残すもの】【変更するもの】【削除するもの】の3つに分ける。削除する場合は
「そのコードが何の役割を持っているか」を1つずつ説明する。役割が分からないコードは削除禁止。

### STEP4：副作用を列挙する
「確実に影響する」「影響する可能性がある」「影響しない」の表でまとめる。

### STEP5：確認済み・未確認を分ける
【確認済み】【コード上そう判断できる】【未確認】の3つに分ける。
推測で「問題ありません」「大丈夫です」とは言わない。

### STEP6：変更案を提示して停止する
「変更前」「変更後」の差分を提示し、ここで止まる。
**ユーザーが「実装してください」と言うまで絶対にコードを書かない。**

### STEP6以降：確認範囲を固定する
STEP6で「実装可能」と判定された後は、新たな確認項目を追加してSTEP1へ戻らない。
今回の修正と直接無関係な確認（例：修正が触れていないブラウザ・OS・状態）は行わない。
確認が完了したら追加調査をせず、「本番へ同期してください」「実装してください」の
指示を待つ。

自分が実際には実行・確認できていない作業を「完了した」ように書かない
（例：ローカルファイルのコピーのみを行った場合に「本番反映しました」と言わない。
ブラウザでアクセスできず未確認の場合は「未確認」と明記する）。
実際に行ったこと／コード上で確認したこと／実機で確認したこと／未確認のことを
常に区別して報告する。

## ①変更前チェック

変更対象だけでなく、そのルール・関数・コンポーネント全体を確認する。

CSSなら
- 削除するプロパティ
- 同じセレクタ内の他プロパティ
- 同じクラスを利用している他箇所

まで確認する。**「不要そうだから削除」は禁止。**

---

## ②削除・変更内容の一覧化（実装前に必ず提示し、ユーザー確認を得てから実装する）

「残すもの」「削除するもの」「削除しても残る機能」を一覧にする。

```
例：.date-row の変更
残す：background-color, appearance:none, padding
削除：flex:1
理由：比率だけ共通コンポーネント側に統一するため
```

---

## ③副作用チェック

変更による副作用を、最低限以下の観点で列挙する。

- 見た目
- JavaScript
- レスポンシブ
- hover
- focus
- ブラウザ差異
- 他ページへの影響

---

## ④共通コンポーネント変更時の確認事項

変更後、そのコンポーネントを使う全ページを一覧化する。

```
例：birthday-input
guardian / zense / kyusei / sansei / numerology / shichu / seiza / sanmei / geimei / rpg
計10ページ
```

---

## ⑤確認済み／未確認の明記

「何を確認したか」だけでなく「何を確認していないか」も必ず書く。

```
確認済み：PC / Chrome拡張機能経由
未確認：iPhone Safari / Android Chrome
```

---

## ⑥推測禁止

「問題ないと思います」のような曖昧な断定は禁止。

- 確認済み
- 未確認
- コード上そう判断できる（推論だが検証はしていない）

を明確に区別して書く。

---

## ⑦削除系変更の差分提示（義務）

削除・置換・共通化を行う場合は、変更前後の差分を「削除されるもの」「残るもの」に
分けて提示する。**この差分をユーザーが確認するまで実装しない。**

目的外のスタイル・処理が変更対象コードに混在している場合は、勝手に削除・変更せず、
理由を説明した上でユーザーに確認を取る。今回の目的以外の変更はしない。

---

## デプロイ前チェック（DEPLOY_CHECKLIST.mdと併用）

- 反映対象ファイル一覧（新規・更新・削除）を確認
- 本番とSTGの実ファイルをdiffし、意図しない差分（別プロジェクトの変更混入等）がないか確認
- 新規ディレクトリ・新規requireファイルの有無を確認
- パス（CSS・画像・リンク）が変わっていないか確認
- コピー順を確認（共通ファイル→各ページファイルの順）
